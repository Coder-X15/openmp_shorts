Initiating a Parallel Section in a C/C++ Program

  • We'll go with the C++ example; the stuff is pretty much the same in C as well - include the omp.h library and type the parallel section in a #pragma omp parallel directive.
    #include <iostream>
#include <omp.h>
.
.
.
int main() {
    #pragma omp parallel{
        // parallelizable section here
        // by default, the max number of threads 
        // are spawned to run in this scope
    }
}
  • We can control the number of threads using omp_set_num_threads():
    #include <iostream>
#include <omp.h>
.
.
.
int main() {
    omp_set_num_threads(4);
    #pragma omp parallel{
        // parallelizable section here
    }
}
  • To print the thread ID/ thread number: use omp_get_thread_num() within the parallel section.

Dealing with Race Conditions

  • Race condition: Sometimes, some threads compete for resources (in the perspective of the timestamp at 4:39, to have their share of processing time). To prevent race condition, one can place those lines of code that may cause a race condition within a #pragma omp critical directive.
    #include <iostream>
#include <omp.h>
.
.
.
int main() {
    #pragma omp parallel{
        #pragma omp critical{
            // critical section here
        }
    
    }
}
  • #pragma omp critical allows only one thread to execute within the critical section at a time - whoever comes first executes first within the scope and then quits the scope, followed by the next thred.
  • Also: #pragma omp single allows the complete execution of only the first thread to reach the section.

Shared and Thread-Private Memory

  • Any variable declared outside the parallelized section is shared, i.e., every thread in the parallelied section can see that address and modify it at will
  • Any variable declared inside the parallelized section is private to that thread alone.
  • To explicitly tell OpenMP which of the variables declared outside the parallel directive is shared and which are private, one can use shared(<vars sep by commas>) and private(<vars sep by commas>) respectively.

    NB: private assigns a local copy of the passed variable to each thread.

  • Eg.:
    #include <iostream>
#include <omp.h>
.
.
.
int main() {
    int x, y, z;
    #pragma omp parallel shared(x, y) private(z){
        // code
    }
}

Parallelizing for loops

  • One can use the #pragma omp for directive within #pragma omp parallel directive or directly use #pragma omp parallel for.

    NB: Do not add #pragam omp critical anywhere inside the loop body; you're gonna make each thread wait and that will again consume some time, making the whole program essentially slow and not fulfil the purpose intended by parallelizing.

  • As an alternative to the point in the nota bene above, you can use a reduction clause alongside the # pragma omp for/#pragma omp parallel for directive.
  • Basically, what the reduction technique does is that it creates thread-local copies of the memory to be shared within the parallelized for loop, lets operations be done on each of it and then in the end operate on the local copies and store the result into the main shared variable.
  • Eg.: A random summation
    #include <iostream>
#include <omp.h>

using namespace std;

int main(){
    int sum = 0; // global, shared variable to the threads
    # pragma omp parallel for reduction(+:sum){
        for(int i = 1; i <= 100; i++){
            sum += i;
        }
    }
    cout << i << endl;
    return 0;
}

  • Run only for the master thread (thread 0): # pragma omp master
  • Get CPU wall time: omp_get_wtime

OpenMP Memory Sharing Clauses:

Clause syntax: clause_name(args)

  • default : specifies default values for code elements when no explicit values are provided. Possible arguments: shared, private (Fortran also has an option none).
  • shared : allows attribs to be shared among multiple threads thus reducing redundancy and increasing code reusability.
  • private : makes variables private to each thread
  • threadprivate : somewhat similar to private ig; check: OpenMP documentation

The below account for the fork-join model in the memory perspective.

  • firstprivate : private copies of that variable for each thread are initialised with the value before the loop iteration; global copy unaffected by change in private copy.
  • lastprivate : special case of private; creates private copies of listed variables but final value is given by the thread that executes last.

OpenMP schedule Clause

  • Follows the parallel for directive (mostly, can be used otherwise as well I guess, but depends on the task)
  • Helps in allocating threads to CPU cores
    • schedule(static, chunk_size) : every chunk_size threads are assigned to each CPU core.
    • schedule(dynamic, chunk_size) : every chunk_size iterations of the loop are assigned to each free CPU core.
    • schedule(guided, chunk_size) : at max chunk_size iterations are given to each free CPU core, but the number of iterations in each chunk may eventually reduce, till the last chunk.
    • schedule(auto) : the system scheduler deals with the thread scheduling.

OpenMP sections Directive

  • A non-iterative work-sharing construct
  • Each section within this directive is assigned to a different thread
  • Syntax (C/C++):
#pragma omp sections [clauses...]
{
    #pragma omp section
    // part of the code to be parallelized - 1

    #pragma omp section
    // part of the code to be parallelized - 2
}

OpenMP barrier Construct

  • Puts an explicit barrier towards the end of the parallelilzed section
  • Helps in synchronizing threads
  • Syntax (C/C++):
#pragma omp parallel{
    // some parallelized code
}
#pragma omp barrier

OpenMP nowait Construct

  • Helps in bypassing implicit barriers within the code
  • Syntax (C/C++) (here, we'll deal with a parallelized for loop):
int n = <some number>;
#pragma omp parallel{
    #pragma omp for nowait
    for(int i = 0; i < n; i++){
        // parallelized code
    }
}

OpenMP num_threads() clause

  • Specifies the number of threads to spawn
  • Usage:
#pragma omp parallel num_threads(<number>){
// parallel section
}

OpenMP if Clause

  • Conditional execution of parallel section
  • Basic syntax:
#pragma omp parallel if(<condition>){
    // parallel section
}

Things to Mind

  • Spawning too many threads needn't help all the time. Spawning of threads itself has a bit of overhead involved, plus Amdahl's law limits the speedup achievable even if you try to parallelize your program massively.
  • For better performance, reduce the number of shared variables.
